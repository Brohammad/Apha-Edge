# Risk Model

This document describes AlphaEdge's pre-trade risk gate, portfolio risk limits, and risk metrics computation.

---

## Pre-trade risk gate

Every order ‚Äî whether submitted via the API or generated by a strategy deployment ‚Äî is evaluated by `RiskGate` before being persisted. If any stage rejects the order, a `RISK_REJECTED` error is returned to the caller.

### Evaluation pipeline

```
ProposedOrder
    ‚îÇ
    ‚ñº
0. kill_switch           ‚Äî platform halt (Redis-backed; admin API)
    ‚îÇ
    ‚ñº
1. position_sizing       ‚Äî quantity > 0, estimated_price > 0
    ‚îÇ
    ‚ñº
2. max_position_exposure ‚Äî projected position weight ‚â§ MAX_POSITION_PCT limit
    ‚îÇ
    ‚ñº
3. portfolio_exposure    ‚Äî projected invested / equity ‚â§ MAX_PORTFOLIO_EXPOSURE_PCT limit
    ‚îÇ
    ‚ñº
4. cash_availability     ‚Äî for BUY: required cash or MIS margin (estimate_required_margin) ‚â§ balance
    ‚îÇ                      for SELL: held quantity ‚â• order quantity
    ‚ñº
5. risk_limits           ‚Äî active MAX_DRAWDOWN and MAX_VAR limits against latest snapshot
    ‚îÇ
    ‚ñº
6. daily_loss_limit      ‚Äî daily P&L loss ‚â§ DAILY_LOSS_PCT of start-of-day equity
    ‚îÇ
    ‚ñº
   ALLOW or REJECT (with stage + reason)
```

### Price estimation

The gate needs an estimated fill price to compute notional and position weights. It uses, in priority order:

1. The order's `limit_price` (if set).
2. The latest daily bar close for the instrument (`BarRepository.get_latest(D1)`).
3. The holding's `current_price` (if the instrument is already held).
4. If no price is available, the gate still runs stages 1, 4 (cash/sell sizing) with `estimated_price=0`, which causes a position-sizing rejection ‚Äî preventing orders with no market data.

## Risk limit types

| Limit type | Description | Unit |
|------------|-------------|------|
| `max_position_pct` | Maximum weight of any single holding in portfolio equity | 0‚Äì1 (e.g. 0.20 = 20%) |
| `max_portfolio_exposure_pct` | Maximum total invested / equity ratio | 0‚Äì1 |
| `max_drawdown` | Maximum drawdown from peak before trading is halted | 0‚Äì1 |
| `max_var` | Maximum 95th-percentile Value-at-Risk (absolute) | Decimal |
| `daily_loss_pct` | Maximum intraday loss as % of start-of-day equity | 0‚Äì1 |

Limits are portfolio-scoped and managed via the `/api/v1/portfolios/{id}/risk-limits` endpoints.

### Default limits (when none are configured)

| Limit | Default |
|-------|---------|
| `max_position_pct` | 100% (no restriction) |
| `max_portfolio_exposure_pct` | 100% (no restriction) |
| Others | No check performed |

Configure limits before deploying strategies to enforce position sizing discipline.

## Platform kill switch

Operators can halt **all** new orders (manual and deployment-generated) without restarting workers.

| Control | Detail |
|---------|--------|
| Storage | Redis key `alphaedge:risk:kill_switch` (shared across API + workers) |
| Gate stage | `kill_switch` ‚Äî evaluated before all other RiskGate stages |
| Admin API | `GET/POST /api/v1/admin/kill-switch` (admin JWT required) |
| Activate | `POST /api/v1/admin/kill-switch/activate` with `{ "reason": "..." }` |
| Deactivate | `POST /api/v1/admin/kill-switch/deactivate` |
| Fail-closed | In `APP_ENV=production`, Redis read failures treat the switch as **enabled** |
| Audit | Activation/deactivation written to `audit_log` |

## Risk metrics (RiskCalculator)

Computed by `RiskCalculator.compute(returns, benchmark_returns, equity_curve)`.

| Metric | Formula | Notes |
|--------|---------|-------|
| VaR 95% | Historical 5th-percentile daily return | Negative = loss |
| VaR 99% | Historical 1st-percentile daily return | |
| Max drawdown | Max (peak ‚àí trough) / peak over equity curve | |
| Sharpe ratio | (mean ‚àí Rf) / std √ó ‚àö252 | Annualised; Rf = 0 |
| Sortino ratio | (mean ‚àí Rf) / downside_std √ó ‚àö252 | Downside only |
| Beta | Cov(port, bench) / Var(bench) | Requires benchmark |
| Alpha | (mean_port ‚àí Œ≤ √ó mean_bench) √ó 252 | Annualised; requires benchmark |
| Volatility | std(returns) √ó ‚àö252 | Annualised |
| Correlation | Pearson r(port, bench) | Requires benchmark |

### Limit enforcer

`LimitEnforcer.check(snapshot, limits, holdings, total_value)` compares a computed `RiskMetricsResult` against configured limits and returns a list of violations. It is used by the portfolio risk endpoint and the daily snapshot Celery task.

---

## Risk snapshots

`RiskSnapshot` records are computed periodically by the `compute_risk_snapshot` Celery task. Each snapshot stores the full metrics result plus a `metrics` dict that can include `start_of_day_equity` ‚Äî used by the daily loss limit check in the risk gate.

**Important:** If no snapshot is available today, the daily loss limit falls back to `portfolio.initial_capital`. Schedule the snapshot task to run daily before market open.

---

## Integration with strategy deployments

When a deployment runner generates a signal, `SubmitOrderHandler._enforce_risk_gate` is called with the deployment's portfolio, holdings, and limits. Rejected orders are logged at `WARNING` level and counted in the `risk_gate_rejections_total` Prometheus counter (labelled by `stage`). The deployment records the rejection as `rejected:<action>` in `last_signal_action`.
